# MuJoCo Conveyor Sorting Demo Plan

## 目的

MuJoCo + Python + OpenCV だけで、ロボットアームがコンベア上を流れる赤・青・黄のブロックをカメラ認識し、色ごとに別の箱へ仕分けるデモを作る。

優先順位は以下の通りとする。

1. クライアント向けに見栄えよく安定して動くこと
2. viewer 上で一連の流れが分かりやすいこと
3. コードが `main.py` から実行できること
4. 物理的な厳密さより、再現性とデモ成功率を優先すること


## 前提方針

- 既存リポジトリにある `unitree_robots/so101/so101_follower.urdf` をアーム候補として流用する
- アーム動作は IK ではなく固定ポーズ列 + 補間で実装する
- 色判定は OpenCV を使い、把持タイミング判定は「カメラ検出結果 + MuJoCo内部座標」のハイブリッドで安定化する
- 把持は完全物理任せにせず、安定デモのために「把持成立後は対象ブロックをグリッパーに追従させる」方式を第一候補とする
- コンベアは本格ベルト物理ではなく、「ブロックを一定速度で搬送するスクリプト制御」を第一候補とする


## 最終成果物イメージ

- `main.py`
  - デモ全体の起動エントリ
- `assets/` または `demo_assets/`
  - MuJoCo シーン XML
- `sorting_demo/` または同等の補助モジュール

  - アーム動作シーケンス
  - フレーム保存
- `output/frames/` など
  - 任意で録画用フレームを書き出す


## 実装ステップ

### Step 1. アーム資産の取り込み方を確定する

最初にやること:

- `so101_follower.urdf` を MuJoCo で単体表示できる既存実装を確認する
- 新規シーンにこのアームをどう組み込むかを決める
- 取り込みが難しい場合の代替案も決める

判断基準:

- 既存の `simulate_python/so101_visualizer.py` の流用が可能か
- シーン拡張のしやすさ
- デモの安定性

代替案:

- URDF 組み込みが煩雑な場合は、簡易な固定ベース6軸アームを MJCF で用意して先にデモ完成を優先する


### Step 2. デモ用シーンを作る

MuJoCo シーン内に以下を配置する。

- ロボットアーム
- 床
- コンベア
- 赤・青・黄色のブロック出現レーン
- 仕分け箱 3 個
- ピック位置の目印
- 仮想カメラ

この段階で決めること:

- ワールド座標系
- コンベアの流れる向き
- ピック位置
- 各色の置き場所
- viewer で見やすいカメラ角度

設計上のポイント:

- ブロックは直方体で統一する
- 箱は色ごとに左右へ十分離して配置する
- ピック位置はアーム可動域の中心付近に置く
- カメラはコンベアを見下ろす角度にして、3色が確実に分離して見える構図にする


### Step 3. Python 側の実行骨格を作る

`main.py` から以下の流れで動く骨格を作る。

1. MuJoCo モデル読み込み
2. viewer 起動
3. デモ管理クラス初期化
4. メインループ開始
5. 毎ステップで物体更新、画像取得、認識、状態遷移、描画保存を実行

想定する主要クラス:

- `DemoController`
- `ConveyorManager`
- `VisionDetector`
- `ArmSequencer`
- `FrameRecorder`


### Step 4. ブロック搬送ロジックを作る

ブロックは 1 個ずつ処理する。

基本仕様:

- 初期位置をコンベア上流に置く
- 一定速度で下流へ流す
- ピック位置に近づいたら停止候補にする
- 1 個の処理が終わるまで次のブロックは流さない

安定優先の実装案:

- ブロック本体は動的物体として置きつつ、搬送中は位置をスクリプトで更新する
- もしくはブロックを kinematic 的に移動させ、停止も位置更新停止で表現する

ブロック生成仕様:

- 色リストを事前定義して順番に流す
- 例: `["red", "blue", "yellow", "red", ...]`
- 毎回 1 個のみアクティブにして状態管理を簡単にする


### Step 5. 仮想カメラ取得と OpenCV 色認識を作る

処理内容:

- MuJoCo のカメラ画像を RGB で取得する
- OpenCV で HSV 変換する
- 赤・青・黄色のマスクを作る
- 輪郭または面積最大 blob を検出する
- 検出矩形、中心点、色ラベルを得る

この認識結果の用途:

- viewer/demo 用の「認識している」見せ方
- 現在流れているブロックの色判定
- ピック位置到達の補助判定

安定化方針:

- 色判定そのものは画像ベース
- 最終的な pick 可否は MuJoCo 内の対象ブロック座標でも確認する
- つまり「見た目はカメラ認識」「動作判定は内部座標でブレを抑える」構成にする


### Step 6. 状態機械を作る

デモ全体を単純なステートマシンで管理する。

想定状態:

- `SPAWN`
- `CONVEY`
- `TRACK`
- `STOP_FOR_PICK`
- `MOVE_TO_PICK`
- `GRASP`
- `LIFT`
- `MOVE_TO_PLACE`
- `RELEASE`
- `RETURN_HOME`
- `RESUME`

状態遷移の基本:

1. ブロック出現
2. コンベア搬送
3. カメラで色検出
4. ブロックがピック位置に来たら停止
5. アームが pick pose へ移動
6. グリッパーを閉じる
7. 箱の上へ移動
8. グリッパーを開く
9. ホームへ戻る
10. 次のブロックを流す


### Step 7. アーム動作シーケンスを作る

IK は使わず、各色共通の pick pose と、色別の place pose を定義する。

必要ポーズ:

- `home`
- `pre_pick`
- `pick`
- `lift`
- `pre_place_red`
- `place_red`
- `pre_place_blue`
- `place_blue`
- `pre_place_yellow`
- `place_yellow`

実装方針:

- 関節角を固定配列で持つ
- 時間ベース補間で滑らかに動かす
- グリッパー開閉も同じ補間枠組みで扱う

ここで最初に確認すること:

- SO-101 の関節順序と可動域
- 現在ある簡易シーケンス例の流用可否


### Step 8. 把持・運搬を安定化する

デモ成功率のため、把持は以下の順で実装を試す。

第一候補:

- ブロックが pick 位置にあり、グリッパーが閉じた時点で「把持成功」とみなす
- 以後、ブロック位置をグリッパー先端 site に追従させる

第二候補:

- MuJoCo の weld/equality を動的に有効化して持ち上げる

この方針にする理由:

- 純粋な接触把持は小型グリッパーだと不安定になりやすい
- クライアントデモでは「毎回成功する」ことが重要


### Step 9. 置き動作と次サイクル再開を作る

配置時の仕様:

- 箱の中心上で一旦停止する
- 少し下げて release pose へ移る
- グリッパーを開く
- ブロックを箱の内部座標へ配置する
- アームをホームへ戻す
- 次のブロック搬送を再開する

簡略化の許容:

- 箱の中へきれいに入ることを優先し、リリース直後のブロック位置を軽く補正してもよい


### Step 10. viewer 表示と録画保存を入れる

viewer で見えるようにするもの:

- シミュレーション本体の動き
- カメラ認識対象が分かる構図

可能なら追加するもの:

- OpenCV の検出結果付きプレビュー画像
- `output/frames/` へのフレーム保存
- 後で動画化しやすい連番 PNG 保存

録画方針:

- 初期版は `--save-frames` オプションで有効化
- 必要なら後段で `ffmpeg` に渡せる形式にする

## 想定リスクと対策

### リスク 1. URDF アームのシーン統合が想定より重い

対策:

- まず単体ロードを再利用する
- 難しければ簡易 MJCF アームへ切り替えて、デモ完成を優先する

### リスク 2. 接触ベース把持が不安定

対策:

- 把持成立後は内部的に追従制御へ切り替える

### リスク 3. 色認識が照明条件でぶれる

対策:

- シーン照明とブロック色を高コントラストにする
- HSV 閾値を固定しやすい配色にする

### リスク 4. コンベア物理が不安定

対策:

- ベルト物理をやり込みすぎず、位置更新ベースの搬送にする


## 着手順まとめ

最初にやることは次の順。

1. SO-101 アームを新規シーンへ載せられるか確認する
2. コンベア・箱・カメラ付きの最小シーンを作る
3. ブロック 1 個を流してカメラ認識する
4. ピック位置で停止させる
5. 固定シーケンスで 1 色だけ pick/place を成功させる
6. 3 色対応に拡張する

## 補足

この計画では、最初から高度な把持や IK を狙わず、確実に見せられるデモを短い手数で完成させる。必要であれば次段階で、把持精度向上、動作自然化、複数物体同時投入へ拡張する。


## 次の改善計画

現在のデモは一連の流れ自体は動いているが、クライアント向けに見せるには以下の不自然さが残っている。

- 赤箱と黄箱の位置が不自然で、レイアウトのバランスが悪い
- ブロックを縦向きに保持しており、見た目が実機らしくない
- アームがコンベア付近を大きく横切るため、すり抜けや干渉に見える場面がある
- 初期姿勢からの移動量が大きく、動き出しが不自然に見える

このため、次の修正は「見た目の自然さ」と「干渉しにくい単純動作」を優先して進める。


### 改善 1. レイアウトを先に作り直す

最初にやること:

- 赤箱、青箱、黄箱をアームの可動域内で左右対称に近い位置へ置き直す
- 特に赤箱と黄箱は、コンベアや土台と視覚的に重ならない位置へ再配置する
- 3 箱とも同じ高さの独立台に載せ、置き姿勢の基準をそろえる

狙い:

- 箱位置の違和感を先に消す
- 各色の place pose を作りやすくする
- 置き動作の軌道を短くする


### 改善 2. 把持姿勢を「横つまみ」前提に変更する

最初にやること:

- pick pose を上からの縦把持ではなく、横方向からの把持に切り替える
- ブロックの保持向きも、持ち上げ中に縦に立たないよう固定する
- pick site 周辺で、グリッパーの向きとブロックの向きが自然に見える組み合わせを決める

狙い:

- 「人が見て普通」に見える把持へ寄せる
- 持ち上げ中の違和感を減らす
- place 時にも箱へ素直に入る向きを保つ


### 改善 3. アームの移動量を最小化する

最初にやること:

- 現在の `home` と `ready` を見直し、最初から pick に近い待機姿勢へ置き換える
- `pre_pick -> pick -> lift -> pre_place -> place -> ready` の最短シーケンスに整理する
- 不要な大振りの回転や遠回りする中間姿勢を削る

狙い:

- 初期の不自然な大きい動きをなくす
- viewer 上でテンポよく見せる
- すり抜けが起きやすい大きなスイングを減らす


### 改善 4. 安全な動作通路を決める

最初にやること:

- アームがコンベア本体の上面や側面を横切らないよう、通ってよい空間を先に決める
- pick 後は一度だけ十分な高さまで持ち上げてから箱方向へ移る
- place 側でも、箱の真上から下ろす単純な経路に統一する

狙い:

- コンベア付近のすり抜けを見えにくくする
- 毎回ほぼ同じ経路を通してデモの再現性を上げる
- 複雑な衝突回避なしで自然に見せる


### 改善 5. 把持後の見た目を安定化する

最初にやること:

- 把持成立後のブロック追従位置を、TCP 直付けではなく「指の間で保持されている」相対位置へ調整する
- 必要なら把持中のブロック姿勢を固定し、箱上で release するまで姿勢を崩さないようにする
- 箱へ置く瞬間だけ drop pose を少し下げ、箱の縁をまたがないようにする

狙い:

- ブロックが指先から浮いて見える問題を防ぐ
- 縦向き保持や途中回転を防ぐ
- place 動作をより自然にする


### 実施順

次の修正は以下の順で進める。

1. 箱の位置と高さを再設計する
2. 横つまみ前提で pick/place pose を作り直す
3. 待機姿勢とシーケンスを短く整理する
4. コンベアを横切らない経路へ統一する
5. 把持中のブロック姿勢と相対位置を微調整する


### この段階の成功条件

以下を満たせれば、次の改善段階は成功とする。

- 赤・青・黄の箱が視覚的に自然な位置に並ぶ
- ブロックを横向きにつまんで運んでいるように見える
- アームがコンベアや箱を大きく貫通して見える場面が大幅に減る
- 初期姿勢から pick/place までの移動量が明らかに小さくなる
