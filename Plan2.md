# MuJoCo Moving Pick Demo Plan

## 目的

現在のデモは「コンベアを止めてから pick する」構成になっている。

次段階では、クライアント向けデモとして見栄えを上げるために、
**コンベアを止めずに、流れているブロックを追従しながら pick する**
デモへ変更する。

優先順位は以下の通りとする。

1. viewer 上で「流れながら掴んでいる」と分かること
2. デモとして毎回安定して成功すること
3. 現在の色認識・色別仕分けの流れを崩さないこと
4. 完全な接触物理より、見た目の自然さと再現性を優先すること


## 現状

現在の実装は以下の流れで動いている。

1. ブロックがコンベア上を流れる
2. ピック位置に来たらコンベアを停止する
3. アームが pick pose に入る
4. グリッパーを閉じる
5. ブロックを追従させて持ち上げる
6. 色別の箱へ置く

この方式は安定だが、デモとしては少し機械的で、テンポも止まりやすい。


## 基本方針

- コンベア停止は行わない
- ブロックは一定速度で流し続ける
- アームは固定ピック点へ行くのではなく、流れている対象物へ短時間追従する
- カメラ認識は「見つけた」演出と色判定に使い、最終的な把持成立判定は MuJoCo 内部座標で行う
- 把持成立後は、現在と同じくブロックをグリッパーへ追従させて安定運搬する

つまり、
**見た目は moving pick、成立判定は内部座標ベース**
の構成にする。


## 実装イメージ

### 1. コンベアは止めない

変更内容:

- `STOP_FOR_PICK` を廃止、または無効化する
- `pause()` を使った停止処理を moving pick 中には呼ばない
- 1 個ずつ処理する方針は維持し、現在の対象ブロックだけを追う

狙い:

- デモのテンポを落とさない
- 「流れ作業感」を出す


### 2. 追従用ステートを追加する

新しい状態の候補:

- `APPROACH_PICK`
- `TRACK_PICK`
- `CLOSE_ON_MOVE`

簡略化するなら、以下のような状態でもよい。

- `SPAWN`
- `CONVEY`
- `TRACK`
- `MOVE_WITH_TARGET`
- `GRASP_ON_MOVE`
- `LIFT`
- `MOVE_TO_PLACE`
- `RELEASE`
- `RETURN_READY`

狙い:

- 「止まって掴む」から「追いかけて掴む」へ状態機械を整理する
- 現在の `place/release/return` の後半部分はなるべく流用する


### 3. pick 姿勢を可変ターゲット化する

現在は `pick_site` が固定位置だが、moving pick では以下の考え方へ変える。

- ブロックの現在位置を毎ステップ取得する
- その位置の少し手前・少し上を追従目標とする
- グリッパーの向きは大きく変えず、横つまみ姿勢を維持する

方針:

- フル IK は使わない
- まずは「横方向の追従量だけを増やした固定シーケンス」にする
- 必要なら `pick/pre_pick` の中間目標を数個用意して、追従窓の中だけ切り替える

狙い:

- 実装を重くしすぎずに moving pick らしく見せる
- 今の安定した横つまみ姿勢を活かす


### 4. 把持成立判定を追加する

moving pick では「閉じた瞬間に必ず掴めたことにする」と不自然になりやすい。
そのため、以下の条件を組み合わせる。

- 対象ブロックと TCP の距離がしきい値以内
- ブロックが pick 窓の範囲内
- 対象色が現在の追跡対象と一致

成立したら:

- その時点で `attach_active_block()` を呼ぶ
- コンベアは止めず、ブロックだけグリッパーへ追従させる
- 以後は現在の `lift -> place -> release` に入る

狙い:

- 見た目のタイミングを自然にする
- 誤把持や空振りを減らす


### 5. 失敗時のリカバリを入れる

moving pick では、対象物が追従窓を抜けることがある。

最低限必要な分岐:

- 把持成立しなかったら、その個体は見送る
- 見送ったら次のブロックをそのまま追う
- 一定回数失敗したら、デモ用に安全な停止式へフォールバックする案も残す

狙い:

- 1 回の失敗でデモ全体が止まらないようにする
- 本番デモ時の安全策を持つ


## 実装ステップ

### Step 1. 現在の停止式 state machine を moving pick 用に分岐する

最初にやること:

- `STOP_FOR_PICK` を使わない流れを作る
- `MOVE_WITH_TARGET` 相当の状態を追加する
- `RETURN_HOME` は `RETURN_READY` 寄りに短くする


### Step 2. 対象ブロック追従ロジックを入れる

最初にやること:

- アクティブブロックの現在位置を取得する
- アームの追従目標を毎ステップ更新する
- 追従量は小さく保ち、横方向中心で動かす


### Step 3. moving grasp の成立判定を入れる

最初にやること:

- TCP と対象ブロックの距離判定を入れる
- 距離しきい値を設定する
- 成立時だけ block attach へ切り替える


### Step 4. 失敗時の見送り処理を入れる

最初にやること:

- pick 窓を通過したのに成立しなかったら失敗扱いにする
- その個体はそのまま流す
- 次個体へ自然に切り替える


### Step 5. デモ速度に合わせて全体テンポを調整する

最初にやること:

- 追従時間
- grasp 判定時間
- lift への切り替え速度
- return の短縮

を再調整する。


## 変更対象の想定

- `sorting_demo/controller.py`
  - 状態機械
  - moving pick 判定
  - 追従処理
- `sorting_demo/poses.py`
  - moving pick 向けの待機姿勢、追従姿勢
- `assets/sorting_demo_base_scene.xml`
  - 必要なら pick 窓の位置を少し手前へ再調整
- `main.py`
  - 必要なら moving pick 用オプション追加


## 成功条件

以下を満たせば初版成功とする。

- コンベアを止めずにブロックを pick できる
- viewer 上で「追いかけて掴んでいる」ように見える
- 3 色とも仕分けの流れを維持できる
- 空振りしてもデモ全体が破綻しない
- 停止式よりテンポが良く、クライアント向けに見やすい


## 想定リスクと対策

### リスク 1. 完全追従にするとアームの動きが不自然になる

対策:

- 追従量を制限する
- 横方向中心で追い、上下姿勢はあまり変えない


### リスク 2. moving grasp の成立タイミングが不安定

対策:

- カメラ認識ではなく内部座標で最終判定する
- 距離しきい値をやや広めに取ってデモ成功率を優先する


### リスク 3. 速度を上げるとすり抜けが増える

対策:

- 把持成立後は現在通り追従へ即切り替える
- 物理接触だけに依存しない


### リスク 4. 失敗時の見栄えが悪い

対策:

- 見送り動作を短くする
- 必要なら失敗時だけ停止式へフォールバックできる構造を残す


## 着手順まとめ

最初にやることは次の順。

1. 停止式 state を moving pick 用に整理する
2. 対象ブロックを追うロジックを入れる
3. moving grasp の成立判定を入れる
4. 失敗時の見送り処理を入れる
5. デモ速度と見た目を調整する


## 補足

この計画では、最初から高度な IK や完全接触把持を狙わず、
**今ある安定デモを moving pick へ拡張する**
ことを優先する。

本当にリアルな moving grasp より、
「クライアントが見て自然で、毎回成功する」
デモを先に完成させる。
